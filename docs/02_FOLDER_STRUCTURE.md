# Folder Structure

## Repository root

```
mbuzi/
├── docs/                 ← this documentation
├── public/                ← unprocessed static files, served as-is
├── src/                   ← everything else
├── astro.config.mjs
├── tsconfig.json
├── package.json
├── CLAUDE.md / AGENTS.md  ← identical content; dev-server instructions for AI coding agents
├── README.md              ← unmodified Astro starter template, not project-specific
└── dist/                  ← build output (gitignored, generated by `npm run build`)
```

`node_modules/`, `dist/`, `.astro/` (Astro's generated types cache), and `.env*` are all gitignored — see `.gitignore`. There is no `.env` file in this project at all (no environment variables are used anywhere).

## `public/`

Files here are copied to the site root **unprocessed** — no optimization, no hashing, served at the exact path they sit at. Only use this folder for files that genuinely need a stable, predictable URL:

```
public/
├── favicon.ico / favicon.svg
├── robots.txt
├── sitemap.xml
├── site.webmanifest
├── images/hero/hero-main.webp   ← default Open Graph image (see below)
└── videos/hero.mp4              ← homepage hero background video
```

Two files deserve explanation:
- **`images/hero/hero-main.webp`** is the fallback Open Graph image (`DEFAULT_SEO.defaultOgImage` in `config/seo.ts`). Social-media link previews need a plain, stable URL — anything routed through Astro's image pipeline gets a hashed build filename (`/_astro/hero-main.HASH.webp`), which an OG meta tag can't reference ahead of time. This is the one exception to "all real photos live in `src/assets/`."
- **`videos/hero.mp4`** — Astro's `astro:assets` image pipeline has no video equivalent, so video files must live in `public/`.

Everything else that used to live under `public/images/*` (backgrounds, events, farm, food, gallery, locations, logos, team) has been removed — those were empty leftover scaffolding from before the project migrated its real photography into `src/assets/`. Nothing in the code references them.

## `src/assets/`

Every real photograph and the site logo, organized by subject:

```
src/assets/
├── astro.svg, background.svg     ← Astro starter leftovers, unused
├── branding/logo.png              ← optional custom logo (falls back to a text/SVG monogram if absent)
├── farm/
│   ├── brand-story.jpg, sustainability.jpg
│   └── gallery/                   ← 6 farm photos (media/farm.ts)
├── food/                          ← ~46 dish photos (data/menu.ts) + food-quality-story.jpg
├── gallery/
│   ├── events/                    ← 4 event photos (media/event.ts)
│   ├── food/                      ← 6 atmospheric grill photos (media/food.ts)
│   ├── locations/                 ← 11 branch interior/exterior photos (media/restaurant.ts)
│   └── family-dining.jpg, outdoor-seating.jpg, private-events.jpg, warm-atmosphere.jpg
├── hero/                          ← one hero image per page (9 total)
└── team/                          ← branch manager + founder portraits
```

**Critical detail**: images are resolved by **filename only**, regardless of subfolder. `src/lib/images.ts` globs every image under `src/assets/**/*.{jpg,jpeg,png,webp,avif}` and builds a lookup map keyed by basename (including extension). A data file's `image: "/images/food/goat-katogo.jpg"` string is never actually fetched as a URL — it's a lookup key, and the resolver finds whatever file is named `goat-katogo.jpg` anywhere under `src/assets/`. This is why folder placement matters for organization but the *filename* is what must match exactly. See [06_MEDIA_SYSTEM.md](./06_MEDIA_SYSTEM.md).

## `src/pages/`

One file per route (Astro's file-based routing). No dynamic routes, no API routes — this is a pure static site:

```
src/pages/
├── 404.astro
├── index.astro          → /
├── menu/index.astro     → /menu
├── farm-story/index.astro → /farm-story
├── gallery/index.astro  → /gallery
├── locations/index.astro → /locations
├── booking/index.astro  → /booking
├── catering/index.astro → /catering
└── contact/index.astro  → /contact
```

Full detail per route in [05_ROUTES.md](./05_ROUTES.md).

## `src/layouts/`

Just `Layout.astro` — the shared document shell (`<html>`/`<head>`/SEO meta/`<body>`). Deliberately does **not** render `Navbar` or `Footer` — every page composes those explicitly, so a page could opt out of either without fighting the layout. It does, however, always mount the two global floating widgets (`CartWidget`, `AssistantWidget`) and the sitewide `FadeIn` scroll-reveal `IntersectionObserver` script, since those genuinely need to be on every page.

## `src/components/`

```
src/components/
├── ui/           ← generic primitives: Button, WhatsAppButton, Logo, OptimizedImage
├── hero/         ← the reusable page hero + its React content/CTA/scroll-indicator
├── cards/        ← FoodCard, LocationCard, TestimonialCard, EventPackageCard, GalleryCard, DeliveryDetails
├── forms/        ← BookingForm + BookingSummary, ContactForm
├── sections/     ← ~30 page-section components (largest folder — see 03_COMPONENT_GUIDE.md)
├── layout/       ← Navbar/NavbarIsland, MobileMenu, Footer
├── animations/   ← FadeIn.astro, the sitewide scroll-reveal primitive
├── assistant/    ← the AI assistant widget's 5 components
└── cart/         ← the WhatsApp ordering system's 4 components
```

The `assistant/` and `cart/` subfolders are organized by **feature**, not by component type — a deliberate difference from the rest of `components/`, because both are self-contained systems with their own supporting `lib/`, `hooks/`, `types/`, `data/`, and `content/` files. Keeping each feature's components together makes the whole system easier to find and remove/replace as a unit.

## `src/content/` vs `src/data/` vs `src/media/` vs `src/config/`

This is the most important structural decision in the codebase. Full explanation with diagrams in [04_DATA_ARCHITECTURE.md](./04_DATA_ARCHITECTURE.md). Short version:

| Folder | Holds | Example |
|---|---|---|
| `src/content/` | Page copy — SEO text, headings, FAQ, CTA labels. One file per page. | `content/menu.ts` → `MENU_CONTENT.hero.headline` |
| `src/data/` | Structured, reusable business data + logic functions. | `data/menu.ts` → `MENU_ITEMS`, `data/delivery.ts` → `getDeliveryInfo()` |
| `src/media/` | Photo/gallery metadata (title, alt text, category) — a specialized subset of "data" split out because it's consumed uniformly by the Gallery page across categories. | `media/farm.ts` → `FARM_IMAGES` |
| `src/config/` | Site-wide facts and design tokens that aren't page-specific. | `config/site.ts` → `SITE.phone`, `config/theme.ts` → `COLORS` |

## `src/types/`

Shared TypeScript interfaces, one file per domain: `content.ts` (the shapes every `content/*.ts` file is built from — `SEOContent`, `HeroContent`, `FAQItem`, etc.), `location.ts` (`Location`, `BranchService`), `media.ts` (`MediaImage` and its category-narrowed aliases), `cart.ts` (`CartLine`, `OrderDetails`), `assistant.ts` (`AssistantMessage`, `RecommendationRule`).

## `src/lib/`

Framework-light utility functions and the two feature-specific subsystems:

```
src/lib/
├── button-variants.ts   ← getButtonClasses() — single source of truth for button styling
├── constants.ts         ← shared Tailwind class strings for form fields
├── helpers.ts           ← isValidName/Phone/Email validators
├── images.ts            ← resolveImage()/resolveImageSrc() — the src/assets lookup system
├── logo.ts              ← LOGO_SRC — optional custom logo resolution
├── cart/                ← CartStore.ts (external store), cartUtils.ts (pure functions)
└── assistant/            ← assistantEngine.ts (AI abstraction), assistantMessage.ts (WhatsApp builder)
```

## `src/hooks/`

Small, genuinely reusable React hooks, each extracted after the same logic was found duplicated in two places: `useBodyScrollLock`, `useFormState`, `useScrollPosition`, `useCart`.

## `src/context/`

Just `AssistantContext.tsx` — the one place in the codebase using React's `createContext`. See [08_AI_SYSTEM.md](./08_AI_SYSTEM.md) for why the cart system deliberately does *not* use Context despite needing similar shared state.

## `src/styles/`

`global.css` — Tailwind's CSS-first entry point (`@import "tailwindcss"`), Google Fonts imports (Inter, Playfair Display), and a small set of legacy CSS custom properties (`--gold`, `--charcoal`, etc.) that are **stale and unused** — no component references them; the real brand colors are the hex literals in `config/theme.ts`'s `COLORS` and hardcoded Tailwind arbitrary values throughout components.
